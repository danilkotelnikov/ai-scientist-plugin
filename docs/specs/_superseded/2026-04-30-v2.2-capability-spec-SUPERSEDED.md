## 1. Goals

v2.2 closes the rigor gap with `Imbad0202/academic-research-skills` (their CC BY-NC license means we **re-implement, never copy**) and lays the linguistic groundwork that v3.0's trained classifier will sit on top of. Six independent capability tracks, each shippable on its own:

| Track | One-line goal | Effort |
|---|---|---|
| **A. Integrity gates (Lu 2026 7-mode)** | Named failure modes M1–M7 fire at explicit pipeline checkpoints | small |
| **B. Typed claim audit (HIGH-WARN classes)** | Replace untyped spot-check with 5-class hard-gate audit | medium |
| **C. Sprint-contract reviewer** | Blind pre-commitment of scoring rubric before reading the paper | small |
| **D. R&R Traceability + stats fallacy + AI disclosure** | Three smaller adds bundled | small-each |
| **E. Russian language first-class output** | ГОСТ-7.0.5 citations, Cyrillic font stacks, impersonal-passive register lint, Russian publisher templates | medium-large |
| **F. Form-driven pre-experimental dialog + post-experiment numerical audit + retrieval-grounded linguistic discriminator + explanatory rationale files** | The "new ideas" net-new to both projects | large |

Total target: ~6–9 weeks of focused work. Tracks A–D parallelize; E and F are the heavyweight items.

## 2. Track A — Lu 2026 7-mode failure checklist

### 2.1 The taxonomy (independent re-implementation; no code copied)

Seven named failure modes that fire at explicit pipeline integrity gates:

| Code | Name | Fires at | Detector |
|---|---|---|---|
| M1 | implementation_bug | post-codegen | `extract_python` AST + pytest smoke + import-resolve check |
| M2 | hallucinated_citation | post-citation-validation | DOI gate + Crossref/DataCite resolve (already in v2.1) |
| M3 | hallucinated_result | post-experiment + pre-manuscript | numerical-claim audit against `results.csv` / `*.npy` (Track F) |
| M4 | shortcut_reliance | post-experiment | check that the metric improved without the claimed mechanism — heuristic: ablation contribution > 50% of total improvement |
| M5 | bug_as_insight | post-experiment | when stderr contains warnings/errors AND results section claims a positive finding |
| M6 | methodology_fabrication | pre-manuscript | check `hypothesis.methodology` field is grounded in cited prior work (each named method has a citation in `references.bib`) |
| M7 | frame_lock | post-reflection-loop | when 3 successive reflection rounds produce ≥80% lexical overlap (the writer is stuck in a phrase loop) |

### 2.2 New module

- `mcp/lib/orchestrator/integrity_gates.py` (~250 LOC)
- Each detector is a function `def m1_implementation_bug(state: PipelineState) -> Optional[IntegrityViolation]` returning either `None` (pass) or a typed violation record.
- `integrity_gates.json` artifact written at end of pipeline: list of every gate that fired, with severity and remediation hint.
- Settings flag: `orchestrator.integrity_gates: ["M1","M2","M3","M4","M5","M6","M7"]` — default all enabled; user can disable specific gates if they conflict with a known-acceptable case.

### 2.3 Acceptance criteria

- `integrity_gates.json` present in every job's output dir
- Each violation has `code` (M1-M7), `severity` (block | warn), `evidence` (the specific signal that triggered), `remediation` (action item for the human)
- Block-severity violations halt the pipeline at the gate; warn-severity continues but raises in the final summary

### 2.4 Tests

- `tests/orchestrator/v2_2/test_integrity_gates.py` — 7 happy-path + 7 negative-path tests (each gate fires when it should, and stays quiet when it shouldn't).

## 3. Track B — Typed claim audit (HIGH-WARN classes)

### 3.1 Five typed warning classes

Replace `cross_validator.stage3_claim_support` (currently an untyped fuzzy-match) with a 5-class typed audit:

| Class | Trigger | Hard-gate? |
|---|---|---|
| `claim-not-supported` | manuscript states "X" with a citation, but the cited paper's tldr/abstract has fuzzy-match-score < 40 against "X" | YES |
| `negative-constraint-violation` | manuscript states "no prior work has X" but a corpus search finds prior work doing X (auto-search OpenAlex + Semantic Scholar) | YES |
| `fabricated-reference` | citation key is in `references.bib` but Crossref/DataCite/OpenAlex all return 404 on the DOI | YES (already in v2.1) |
| `anchorless` | a quantitative claim (number, percentage, date) appears in the manuscript without an inline citation within 200 chars | WARN |
| `constraint-violation-uncited` | the methodology section mentions a constraint or limitation that is not cited to any prior work | WARN |

### 3.2 Calibration

Gates fire on a per-claim basis. Calibrate against a labeled gold set:

- FNR target < 0.15 (we miss < 15% of true positives)
- FPR target < 0.10 (we falsely flag < 10% of valid claims)

Gold-set construction: 30 manuscripts × 5-10 claims each = ~200 labeled examples. User-supplied or community-built (the latter belongs in v3 product surface).

### 3.3 New module / changes

- Extend `mcp/lib/orchestrator/cross_validator.py` with `stage3_typed_claim_audit(manuscript_tex, references_bib, papers_metadata) -> List[ClaimViolation]`.
- Pull `tldr` field from Semantic Scholar (already fetched in Stage 2; just read it).
- For `negative-constraint-violation`, dispatch a one-shot OpenAlex search for the negated topic.
- Emit `claim_audit.json` instead of the current `claim_support_matrix.md` (which we'll keep as a human-readable rendering).

### 3.4 Acceptance criteria

- Every quantitative claim in the manuscript is annotated with one of: validated, anchorless (warn), claim-not-supported (block), fabricated-reference (block), or negative-constraint-violation (block).
- Pipeline halts on block-severity claims unless `orchestrator.cross_validation.allow_unverified = true`.
- `claim_audit.json` has FNR/FPR thresholds + actual classified counts.

## 4. Track C — Sprint-contract reviewer (blind pre-commitment)

### 4.1 Two-phase reviewer protocol

Replace single-pass reviewer dispatch in `phase_7_review` (and `phase_7r_review`) with two phases:

**Phase 7.1 — paper-blind rubric pre-commitment.** Each reviewer (3-bias: positive / negative / neutral) is dispatched WITHOUT the manuscript and asked to:
- Declare the dimensions they will score (e.g., "novelty 1-5, methodology 1-5, results 1-5")
- Declare what counts as a failing score on each dimension
- Declare what evidence in the paper would change their mind
- Output: `reviewer_pre_commitments.json` (one entry per reviewer)

**Phase 7.2 — paper-visible review against committed rubric.** Each reviewer is then given the manuscript + their own Phase 7.1 commitment. They must score per the committed rubric, not re-derive new criteria. Deviations from the commitment are tracked and surfaced.

### 4.2 Anti-anchoring guarantee

If a reviewer in Phase 7.2 wants to add a new dimension not in their Phase 7.1 commitment, they must explicitly justify the deviation (e.g., "I committed to scoring on novelty + methodology + results but the paper makes a strong claim about reproducibility that I want to score on"). The justification lands in `reviewer_deviations.json` for transparency.

### 4.3 New artifacts

- `reviewer_pre_commitments.json` — 3 entries (one per bias)
- `reviewer_deviations.json` — empty in the happy path; populated when reviewers add dimensions mid-review
- `review.json` — existing artifact, now annotated with which dimensions were pre-committed vs added

### 4.4 Acceptance criteria

- Every entry in `review.json` cross-references a `reviewer_pre_commitments.json` rubric entry OR a `reviewer_deviations.json` justification.
- The aggregator (median Overall, IQR, consensus_high) only counts dimensions that all 3 reviewers either pre-committed to or deviated to with justification.

## 5. Track D — R&R Traceability + stats fallacy + AI disclosure (bundled)

### 5.1 R&R Traceability Matrix (revision tracking)

When a pipeline is re-run with `--resume` after a review pass:

- For each major reviewer finding, the pipeline auto-generates a row in `rr_traceability.md` with columns: `reviewer`, `finding`, `severity`, `author_claim` (what the writer says they addressed), `verified` (whether a diff against the prior manuscript actually changes the relevant section).
- New module: `mcp/lib/orchestrator/revision_tracker.py` (~120 LOC).

### 5.2 11-type statistical fallacy detector

In `experiment-runner` (post-execution), scan `results.csv` and the experiment's output `stdout.txt` for 11 named fallacies:

1. Simpson's Paradox (subgroup directionality flip)
2. Survivorship bias (filtered population)
3. Selection bias (non-random sampling)
4. Confounding (unmeasured variable correlation)
5. Multiple-comparison without correction
6. p-hacking (effect found only at one of N alpha thresholds)
7. Cherry-picking (best-of-N seed reporting)
8. Regression to the mean
9. Extrapolation beyond support
10. Reverse causation
11. False precision (3-decimal accuracy reported on n=5 data)

Each detector is a heuristic, not a guarantee. Emit `stats_fallacies.json` with detected risks + severity.

### 5.3 Venue-specific AI disclosure generator

New slash command: `/ai-scientist-disclosure <venue>`. Outputs the venue's required AI-usage disclosure boilerplate, populated with the actual orchestrator usage stats from the run.

Venues to support in v2.2: NeurIPS, ICML, ICLR, ACL, EMNLP, Nature, Science, ACS, Cell. (Each has its own boilerplate string.)

Implementation: a YAML registry at `templates/ai_disclosure/{venue}.yaml` with the venue's required wording template. Token-substitute the actual orchestrator usage (models used, hosted vs local, BYOK status, etc.) and output to `ai_disclosure.md`.

### 5.4 Acceptance criteria

- `rr_traceability.md` written on every `--resume` run, populated with auto-verified revisions
- `stats_fallacies.json` written by experiment-runner, with named-fallacy detections
- `/ai-scientist-disclosure neurips` (or any of the 9 venues) outputs valid disclosure prose

## 6. Track E — Russian language first-class output

### 6.1 ГОСТ-7.0.5 citation engine

Russian academic style: ГОСТ Р 7.0.5-2008 ("Библиографическая ссылка"). Differs from APA/Chicago/IEEE in:

- Author surnames first, initials after, with periods: `Иванов И. И.`
- Long-dash separator between authors and title: `Иванов И. И. Название работы`
- Year in `// Журнал` block after title
- Russian abbreviation conventions: `c.` for pages (not `p.`), `вып.` for volume, `№` for number
- Cyrillic OR Latin orthography preserved per source (we don't transliterate)

Implementation:
- New module: `mcp/lib/orchestrator/citation_styles.py` (~300 LOC) — table-driven renderer with backends for APA, Chicago, IEEE, MDPI, **ГОСТ-7.0.5**, ACS, Vancouver, Harvard.
- New CLI flag: `--citation-style gost` (or `apa`, `ieee`, etc.). Default per article-type (review-articles in Russian: gost; experimental in English: ieee).

### 6.2 Russian academic prose register

Russian academic register is different from English:

- Heavy use of impersonal passive (`было показано`, `можно отметить`) — but NOT to the point of dropping agent attribution
- Formal lexicon, no contractions
- No first-person plural authorial "we" (use impersonal: `в настоящей работе показано`)
- No paragraph-initial connective particles (`Кроме того`, `Более того`, `Также` — these are the Russian equivalents of our blacklisted "Furthermore"/"Moreover")

`anti_llm_lint.py` extended with a `lang="ru"` mode:
- Russian-equivalent Tier 1-3 blacklist (curated from Russian academic writing guides)
- Em-dash density ceiling tuned for Russian (Russian academic prose uses dashes more liberally than English; budget ~4 / 1k words instead of 2)
- New paragraph-initial particles to flag: `Кроме того`, `Более того`, `Также`, `Стоит отметить`, `Важно подчеркнуть`, `Следует отметить`

### 6.3 Russian publisher templates

In v2.2, add LaTeX templates for the three highest-impact Russian venues:

- **ВАК-perechen' generic journal** (multiple disciplines)
- **«Доклады Российской академии наук»** (RAS reports)
- **«Успехи современной науки»** (catch-all)

Word output: `.dotx` templates derived from the venues' submission guidelines.

Files:
- `plugins/ai-scientist/templates/latex/gost-generic/`
- `plugins/ai-scientist/templates/latex/dan-ras/`
- `plugins/ai-scientist/templates/latex/uspekhi/`
- `plugins/ai-scientist/templates/word/gost-generic.dotx` (derived from public manuscript guidelines; no copyrighted content embedded)

### 6.4 Cyrillic font stack

Tectonic (our LaTeX compiler) handles Cyrillic with `\usepackage[utf8]{inputenc}\usepackage[T2A]{fontenc}` + appropriate font (we recommend `CMU Serif` or `Noto Serif`). Add to all Russian templates.

For Word output: embed the relevant font into the `.dotx` styles (`Times New Roman` for ГОСТ, `Liberation Serif` for portable rendering).

### 6.5 Acceptance criteria

- Running `/ai-scientist <topic> --lang ru --citation-style gost --venue dan-ras` produces a fully ГОСТ-compliant Russian manuscript with correct Cyrillic typography in both PDF and DOCX.
- `anti_llm_lint.py --lang ru` flags Russian-specific LLMishms (e.g., paragraph-initial `Кроме того,`).
- `references.bib` entries preserve their original-language orthography (no transliteration of Russian-language sources).

## 7. Track F — The new-ideas net-new bundle

Four features that neither Imbad nor we currently have. These are the v2.2 differentiators.

### 7.1 Form-driven pre-experimental dialog

New slash command: `/ai-scientist-setup` (or first-invocation prompt when running `/ai-scientist <topic>` without prior setup).

A structured form collects, BEFORE any agent dispatches:

1. Research question (free-form)
2. Domain (one of 8 niches; auto-pre-fills domain templates)
3. Article type (review / experimental / benchmark / theoretical / case-study / policy-brief)
4. Target venue (one of the registered venues; affects citation style, length budget, format)
5. Output language(s) (one or more of: EN / RU / + planned languages)
6. Compute budget (max external API requests; max wall-clock)
7. Codebase path (optional — triggers codebase-scanner)
8. Existing literature corpus (optional — path to Zotero JSON / folder of PDFs / Obsidian vault)
9. Author list + ORCID + affiliations
10. Conflict-of-interest declarations
11. Funder acknowledgements

Output: `job_config.json` — the full pipeline configuration, deterministically routable to article-type-specific phase order.

Implementation: `phase_minus_2_setup` runs an interactive `AskUserQuestion` cascade. The `AskUserQuestion` tool's structured-response format is perfect for this. Maximum 4 questions per round; spread across 3 rounds.

### 7.2 Post-experiment numerical-claim audit

After `phase_5r_manuscript_review_article` and before `phase_6_citations`:

- Parse the manuscript's results section (using a simple LaTeX → AST parser for tables + figures + body claims)
- Extract every quantitative claim: numeric values, percentages, accuracy/loss/F1/AUC reports
- For each: look up the source artifact (`results.csv` for table values; figure caption + rendered PNG for figure claims; `experiment_stdout.txt` for inline numbers)
- Compute `claim_value` vs `artifact_value` for each pair
- Tolerance: 1e-3 absolute / 1% relative (configurable)
- Mismatches go into `numerical_audit.json` with `claim`, `claim_value`, `artifact_value`, `artifact_path`, `delta`, `severity`

New module: `mcp/lib/orchestrator/numerical_audit.py` (~250 LOC).

### 7.3 Retrieval-grounded linguistic discriminator

v2.2 ships the **retrieval-grounded** version (corpus + k-NN). v3.0 layers the trained classifier on top.

Steps:

1. **Curate the corpus.** Per-discipline (8 niches × 150 papers ≈ 1,200 OA papers from arXiv / PubMed Central / DOAJ / OpenAlex OA). Filter: must be OA full-text, English OR Russian, peer-reviewed venue. **A single automated script** at `scripts/curate_corpus.py` does the harvest. Output: per-paragraph chunks with metadata (discipline, venue, year, language) in MemPalace.

2. **Index.** ChromaDB collection per discipline, embedding model: `intfloat/multilingual-e5-large` (handles both EN + RU). Stored at `~/.ai-scientist/corpus/{discipline}/chroma.db`.

3. **Runtime check.** During `anti_llm_lint`, for each paragraph the manuscript writer produces:
   - Embed the paragraph
   - k-NN against the discipline-matched corpus (k=5)
   - Score: cosine similarity to the closest 5 paragraphs
   - If score < threshold (default 0.55): flag as "out-of-register" — paragraph doesn't read like the discipline's published corpus

4. **Judge.** When a paragraph is flagged out-of-register, dispatch a small LLM judge (sonnet, low cost) with the paragraph + 3 retrieved corpus examples. Judge says: pass (false positive) / fail-rewrite / fail-rewrite-with-suggestion.

5. **Loop.** Failed paragraphs go back to the manuscript writer with the corpus exemplars inlined as `<input name="register_exemplars">`. Max 2 retry rounds per paragraph.

Files:
- `scripts/curate_corpus.py` — single automated script (user can run `python scripts/curate_corpus.py --discipline all --language en,ru` to provision)
- `mcp/lib/orchestrator/register_discriminator.py` — runtime k-NN + judge
- Settings: `orchestrator.writing_style.register_discriminator: { enabled: bool, threshold: 0.55, max_retries: 2 }`

### 7.4 Explanatory rationale files

For every artifact the pipeline writes (manuscript, plots, code, references), also write a `.rationale.md` companion file explaining:

- Why this artifact exists (which research question / hypothesis it serves)
- What decisions the agent made (which competing options were considered; which was chosen and why)
- What evidence the agent relied on (which papers; which numerical values; which prior work)
- What the human researcher should verify (3-5 specific check-this items)
- What the agent is uncertain about (open questions / hedges)

Implementation: each major agent (ideator, hypothesizer, code-generator, manuscript-writer, plotter, reviewer) is augmented with a final "write rationale" output that feeds a small `rationale-writer` agent. The rationale-writer consumes the agent's decision log + key inputs and writes the markdown file.

Files:
- `plugins/ai-scientist/agents/rationale-writer.md` (new agent)
- Hooks into existing agents' output schemas to capture decision logs

### 7.5 Acceptance criteria

- `/ai-scientist <topic>` with no prior setup triggers `/ai-scientist-setup` once and produces `job_config.json`
- Every job's output has `numerical_audit.json` with claim ↔ artifact value comparison
- `register_discriminator.json` records per-paragraph corpus-similarity scores; flagged paragraphs were retried
- Every major artifact has a `.rationale.md` companion
